1 # The IDA-LET-Machine (ILM)
2 The ILM is a simple implementation of the logical execution time (LET) paradigm for the automotive domain.
3 In general it is based on a double buffering, to implement a zero-time communication between different cores.
4 You can find the doxygen generated documentation [here](https://matthiasb85.github.io/IDA-LET-Machine/files.html "ILM documentation")
5 
6 The ILM based was developed as an operating system (OS) and hardware independent module.
7 At the moment, we support the standard OSEK/AUTOSAR OS API as the first OS port was based on ERIKA OS.
8 The current hardware port is based on the CCU6 module of the AURIX TC2xx which is usually used for PWM generation.

14 ## Where is the ILM intended for?
15 Our mission is, to provide a simple and efficient test framework for the LET paradigm.
16 This framework can be used in academia (e.g. for evaluation) or also for prototyping.
17 In general the ILM can be used everywhere, where you would like to test something new, but don't want to have a huge ammount of 3rd party software in the background.
18 
19 ## Where is the ILM not intended for?
20 Although the license model does not exclude it explicitly, we would recommend not to use the ILM in any kind of series implementation.
21 We have written the software to the best of our knowledge and belief, nevertheless it may still contain some bugs.
22 Therefore we can not provide any kind of guarantee for functional correctness.
23 For this kind of integegration there are several other commercial partners who provide their own implementation.
24 As an example, Bosch and Continental do both provide an implementation of their interpretation of the LET paradigm.

31 ## Where to start?
32 In order to test the example, you need a build environment for the AURIX microcontroller.
33 We use a Linux in combination with the HighTech tricore-gcc, but a cygwin installation (as used for ERIKA OS on a Windows) should also do the work.
34 
35 ```
36 git clone https://github.com/matthiasb85/IDA-LET-Machine.git
37 cd ./IDA-LET-Machine
38 cd Example
39 make
40 ```
41 The Makefile checks if the 3rdParty software is available, if this is not the case the ./3rdParty/Setup3rdParty.sh script is executed.
42 The script downloads the corresponding standalone version of ERIKA OS and RT-Druid, and applies some patches on it.
43 The hardware port for the AURIX microcontroller uses Infineons iLLD library, which unfortunately is only available after registration.
44 If the library is not available in ./3rdParty/Download when ./3rdParty/Setup3rdParty.sh is called, the script reports:
45 
46 ```
47 Can't find Infineon iLLD!
48  1.) Please get in contact with Infineon
49  2.) Download the AURIX_iLLD_Package_TC2xx_v1.0.1.3.0 driver package
50  3.) Copy AURIX_iLLD_Package_TC2xx_v1.0.1.3.0.zip to ./3rdParty/Download
51  4.) Run 'make'/'make prepare' in ./Example or Setup3rdPartySoftware.sh in ./3rdParty
52 ```
53 An important constraint is, that the entire library is available as source code.
54 The reason for this is a bug inside the official CCU6 driver of the iLLD (which is patched during ./3rdParty/Setup3rdParty.sh).

56 ## Application example
57 The application consist of 18 LET tasks.
58 The layout of the example application can be found in ./Example/ilm_example.ods
59 The example uses a setup with three application cores.
60 As the used AURIX microcontroller provides three cores, we use core C0 to generate the needed IRQ pattern.
61 In addition, the application tasks are executed on all three cores.
62 The worst-case execution time (WCET) of the corresponding block is assigned to an LET task on both application cores.
63 If no function block is assigned to an LET task on C0, C1 or C2, the corresponding WCET is zero.
64 In order to simulate an accurate timing behavior of the function blocks, we used an optimized **nop** loop consuming CPU time.
65 Due to the different periods of the system, the hyperperiod is 100ms.
66 
67 During execution of the simulated function blocks, we use pin toggles to indentify the current states.
68 Those pin toggles are issued through the ILM hooks and at execution begin/end of a function block.
69 The physical pin magging is located in ./Example/IO/gpio.c and the hook functions in ./Example/ILM/ilm_hooks.c
70 
71 If you use a Lauterbach Trace32 based setup, you can use the ''make debug'' command inside ./Example to start a debug session.
72 With ''t32_folder'', ''t32_arch'' and ''addr'' you can pass Trace32 base folder, host CPU architecure and IP address (DNS name) of your trace hardware to the system.
73 If you don't pass a parameter for ''addr'', Trace32 will search for an USB coupled device.
74 As an example, the ''make debug'' call on my PC looks like this:
75 ```
76 make debug t32_folder=/home/matthiasb/opt/t32/files t32_arch=pc_linux addr=lauterbach1
77 ```
